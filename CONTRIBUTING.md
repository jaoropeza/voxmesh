# Contributing to VoxMesh

Read [AGENTS.md](AGENTS.md) first — it is the primary development contract for both human and
AI-assisted contributions. This document covers the mechanics.

## Workflow

Trunk-based development via GitHub Flow (ADR-0017):

1. Every change starts from a GitHub issue with acceptance criteria.
2. Branch from `main` using `<type>/<issue-number>-<short-description>`, e.g.
   `feat/123-wasapi-loopback`. Types: `feat`, `fix`, `test`, `docs`, `chore`, `refactor`.
3. Keep branches short-lived; rebase or update before merge.
4. Open a draft pull request early. Fill in every section of the PR template.
5. `main` must always be releasable. No direct pushes; squash merge; delete the branch after merge.
6. Use feature flags for incomplete product behavior — never merge dead half-features without one.

## Pull request expectations

- Under ~500 changed lines excluding generated files, fixtures, lock files, and mechanical
  formatting. Larger PRs need a written reason and a review plan in the description.
- One approval for ordinary changes; two for security, authentication, encryption, media capture,
  real-time audio, contract, or persistence changes (see `docs/development/branch-protection.md`).
- Material architectural decisions require an ADR in `docs/adr/` (format in the master prompt §25)
  before or alongside the implementation.
- Declare AI assistance in the PR template section provided.

## Quality gates

All CI checks must pass. Per-language commands are in the [README](README.md). Never bypass tests
or reduce coverage because code was generated by an AI.

## Commit style

`<type>: <imperative summary>` — e.g. `feat: add bounded ring buffer for capture frames`.
Reference the issue in the PR, not necessarily in each commit.
